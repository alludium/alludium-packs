Technical Evaluation Guide

Lightweight technical evaluation guidance for product, architecture, technical edge, IP, data, and roadmap hypotheses.

When this source is used as a rendered task output, create the final deliverable as a standalone safe static HTML artifact with a .html filename and mimeType: "text/html". Use the shared VC document styling rules from Template Use Guidance and make any generated links open in a new tab or window with target="_blank" and rel="noopener noreferrer". Do not create Markdown output artifacts from this source.

Purpose

Use this guide to evaluate technical plausibility and defensibility before formal diligence. The output should identify the technical claims that matter most to the investment case and what evidence is needed to verify them later.

Core Questions

AreaEvaluation QuestionEvidence To Seek
ProductWhat has been built and what is still roadmap?Demo notes, product screenshots, deck, release notes, customer usage.
Technical edgeWhat is the claimed technical advantage?Architecture notes, model/data claims, benchmarks, patents, founder explanation.
ArchitectureDoes the technical approach seem plausible and scalable?System overview, infrastructure notes, integration complexity, team explanation.
Data and AIAre data sources, model claims, latency, rights, and evaluation methods credible?Data-source summary, model results, validation method, compliance notes.
IP and defensibilityIs the edge protected by IP, data, process, switching cost, or know-how?Patent claims, trade-secret posture, assignments, proprietary data, workflow lock-in.
RoadmapCan the roadmap support commercial milestones?Milestone plan, engineering capacity, dependencies, product gaps.
Team coverageDoes the team have enough technical leadership?Founder/team bios, CTO/technical lead role, hiring gaps.

Output Standard

SectionRequired Content
Current technical viewPositive / mixed / negative / unknown.
Main technical thesisThe technical belief that must be true.
Technical claims registerClaim, evidence, confidence, and proof gap.
Gating risksRisks that could block decision review or formal diligence.
Next proof neededDemo, architecture review, expert call, model/data evidence, IP documents, or code-access plan.
Decision impactWhether the opportunity should continue, pause, or require targeted technical diligence.

Boundary

Technical evaluation is not a code review, security audit, expert technical diligence, or IP clearance. State when those inputs are still required.
